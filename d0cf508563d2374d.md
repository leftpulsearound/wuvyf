# LinkPedia 技术资源导航站

LinkPedia 是一个面向开发者、技术研究人员与开源爱好者的结构化外链与知识索引系统。项目定位为技术资源导航站，核心目标是对分散于互联网各处的优质技术文档、百科条目、开发教程与工程实践文章进行系统化采集、分类与呈现，帮助用户快速定位所需信息，减少重复检索成本。

项目不生产原始内容，而是通过人工筛选与自动化采集相结合的方式，建立高质量外部链接的索引目录。所有资源均经过基础可用性校验与主题归类，支持按技术领域、内容形式、更新时间等多维度检索。LinkPedia 适用于个人知识管理、团队技术选型参考、开源项目文档聚合以及技术写作素材收集等场景。

## 功能概览

**多维度分类索引**：按编程语言、框架生态、基础设施、算法理论、工程实践等一级分类组织链接资源，每个分类下细分二级标签。

**链接可用性监控**：定期对收录的 URL 进行 HTTP 状态检查，自动标记失效链接并生成报告，确保索引库的活性。

**全文元数据提取**：对每个收录链接自动提取页面标题、描述、关键词及主要章节结构，生成摘要信息供快速预览。

**自定义标签与收藏**：用户可为任意链接添加自定义标签与备注，构建个人化的资源子集，支持私有与公开两种可见性。

**批量导入与导出**：支持通过 CSV 或 JSON 格式批量导入链接清单，导出功能支持 Markdown 表格、JSON 及 OPML 格式。

**RSS 订阅与更新通知**：基于链接历史变化生成 RSS 订阅源，当目标页面发生内容更新时主动推送通知。

## 应用场景

技术团队知识库建设：团队 Leader 或技术负责人可将 LinkPedia 作为内部文档系统的外链补充，将分散在各处的官方文档、最佳实践文章、故障排查记录统一收录，新成员入职时可快速了解团队所依赖的技术生态全貌。

个人开发者日常查阅：全栈开发者在日常工作中频繁查阅各类框架文档、API 参考与算法实现，通过 LinkPedia 建立个人化的快速访问面板，将高频使用的资源集中管理，避免每次从搜索引擎重新检索。

技术写作与内容创作：技术博主或开源项目维护者在撰写技术文章或项目文档时，需要引用大量外部资料作为参考依据。LinkPedia 的批量导出功能可一键生成参考文献列表，大幅提升写作效率。

开源项目生态导航：开源项目维护者可在项目 README 中嵌入 LinkPedia 索引页链接，为社区贡献者和用户提供外围资源导航，包括相关工具、插件、教程及案例实践，降低生态系统的信息不对称。

## 快速开始

以下命令可在本地环境中完成 LinkPedia 站点的克隆、依赖安装与开发服务器启动。

```bash
git clone https://github.com/linkpedia/linkpedia.git
cd linkpedia
npm install
npm run dev
```

生产环境构建与启动：

```bash
npm run build
npm start
```

Docker 方式部署：

```bash
docker build -t linkpedia:latest .
docker run -p 3000:3000 linkpedia:latest
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Node.js >= 20.0.0 | 是 | 运行时环境，推荐使用 LTS 版本 |
| npm >= 10.0.0 或 yarn >= 1.22.0 | 是 | 包管理器，用于安装项目依赖 |
| PostgreSQL >= 15.0 | 是 | 主数据库，存储链接元数据与用户数据 |
| Redis >= 7.0 | 是 | 缓存层与任务队列，用于链接监控与通知 |
| Elasticsearch >= 8.0 | 否 | 全文检索引擎，未安装时降级为 SQL 模糊查询 |
| MinIO / S3 兼容存储 | 否 | 存储导入的 CSV/JSON 文件及导出产物 |
| PM2 >= 5.0 | 否 | 生产环境进程守护，推荐用于持久化运行 |

## 项目结构

```
linkpedia/
├── apps/
│   ├── web/                         # 主 Web 应用（Next.js 14，App Router）
│   │   ├── app/                     # 页面路由与布局
│   │   ├── components/              # 可复用 UI 组件（链接卡片、分类树、搜索框）
│   │   └── hooks/                   # 自定义 React Hooks（useLinkMonitor、useSearch）
│   └── worker/                      # 后台任务服务（Node.js + BullMQ）
│       ├── queues/                  # 队列定义（监控队列、导入队列、导出队列）
│       └── processors/              # 任务处理器（链接检查、元数据提取、RSS 生成）
├── packages/
│   ├── db/                          # 数据库层（Prisma Schema + 迁移脚本）
│   │   └── prisma/                  # schema.prisma 与 migrations/
│   ├── api/                         # 公共 API 类型定义与 Zod 校验
│   │   └── types/                   # 请求/响应类型、枚举常量
│   ├── crawler/                     # 链接采集引擎（基于 Puppeteer + Cheerio）
│   │   ├── fetcher/                 # HTTP 请求封装与重试策略
│   │   └── parser/                  # 页面元数据解析器（title、description、headings）
│   └── shared/                      # 工具函数与常量
│       ├── validators/              # URL 校验、分类标签校验
│       └── formatters/              # 日期格式化、文件大小转换
├── configs/
│   ├── eslint/                      # ESLint 配置（Monorepo 统一规则）
│   ├── typescript/                  # tsconfig.json 基础配置
│   └── jest/                        # 单元测试配置
├── docker/
│   ├── docker-compose.yml           # 本地开发全栈编排（PostgreSQL + Redis + ES）
│   └── Dockerfile.web               # Web 应用生产镜像
├── scripts/
│   ├── seed.ts                      # 初始分类与演示数据填充
│   └── healthcheck.sh               # 依赖服务健康检查脚本
├── tests/
│   ├── unit/                        # 单元测试（Jest）
│   └── e2e/                         # 端到端测试（Playwright）
├── .env.example                     # 环境变量模板
├── package.json                     # 根 package.json（workspaces 定义）
├── turbo.json                       # Turborepo 流水线配置
└── README.md                        # 项目入口文档（本文件）
```

## 贡献指南

## 常见问题

问：LinkPedia 与普通浏览器书签管理器有何区别？

答：浏览器书签管理器仅提供基础的 URL 存储与文件夹分类，缺乏自动化元数据提取、链接可用性监控、全文检索、批量导入导出以及协作共享能力。LinkPedia 定位于团队级知识索引系统，强调资源的可发现性与状态可观测性，而非个人收藏夹的替代品。

问：如何添加自定义分类或标签？

答：管理员可在后台管理界面的分类管理模块新增一级分类或二级子分类。普通用户可为每个链接添加最多 10 个自定义标签，标签在用户个人空间内私有，不影响全局分类体系。标签支持模糊搜索与批量筛选。

问：链接监控的频率是多少？是否会触发目标网站的访问压力？

答：默认监控周期为每 7 天一次，每次检查仅发送单次 HEAD 请求，不下载完整页面内容，对目标服务器造成的负载可忽略。监控时段固定为 UTC 时间凌晨 02:00-05:00，避开目标网站的业务高峰。用户可在配置文件中调整监控间隔或完全关闭自动监控。

# ResourceLink Hub

ResourceLink Hub 是一个面向开发者、技术研究人员与信息分析人员的结构化外链资源聚合平台。本项目不生产内容，而是对互联网上已有高价值信息页面进行人工筛选、分类归档与索引呈现，帮助用户快速定位特定主题下的参考文章、百科词条与技术文档。

项目定位为“技术资源的导航站”，主要解决如下问题：信息分散导致查找效率低下、搜索引擎返回大量低质或无关结果、跨站点资源缺乏统一索引入口、以及知识沉淀过程中缺乏可分享的链接清单。ResourceLink Hub 通过按批次组织链接清单，配合清晰的目录结构与文档导航，使资源可检索、可追溯、可扩展。

本项目第 4 批次收录 180 个外链资源，涵盖通用百科、技术词条、行业术语解释及信息参考页面。所有链接均保留原始 URL 格式，不做重定向、短链替换或内容抓取，确保指向源站的最新信息。

## 功能概览

**按批次组织资源清单**：每一批次对应一个独立的资源集合，批次内链接按原始顺序排列，支持增量更新与版本追溯。

**保留原始 URL 地址**：所有链接以纯文本形式原样呈现，不添加任何追踪参数、跳转中间页或格式化包装，确保指向的准确性。

**多维度分类索引**：资源按内容主题、文件类型或来源站点进行逻辑分组，便于按类别浏览。

**文档导航表格**：提供面向不同角色的导航路径，包括维护者、终端用户、API 调用方和数据分析人员，每种角色对应不同的目录入口。

**项目结构可视化**：以 ASCII 目录树展示项目文件组织方式，包含脚本、配置、数据、文档和测试等子目录，每行附带功能注释。

**轻量化快速部署**：基于静态站点生成机制，仅依赖标准 Python 环境与基础依赖库，三步完成从克隆到运行。

**贡献流程规范化**：定义清晰的外部资源提交流程，包括链接审核、分类标注、格式校验和批次合并。

**常见问题自助排查**：内置高频问题解答，覆盖链接访问异常、分类归属疑问和批次更新机制。

## 应用场景

技术团队内部知识库建设：技术负责人可以将 ResourceLink Hub 作为团队知识库的外链索引模块，每批次收录的链接对应一个技术专题，例如第 4 批次可对应“网络协议”、“数据库原理”或“编程语言特性”等方向。团队成员通过浏览链接清单快速获取外部参考材料，减少重复搜索时间。

个人开发者的学习路径管理：开发者可将本项目作为个人学习清单的托管载体，每批次链接对应一个学习阶段。例如，学习 Rust 语言时，将官方文档、博客教程和最佳实践文章按批次收录，配合项目结构中的脚本实现链接可用性检查，确保学习过程中参考资料始终可访问。

数据采集与内容分析的前置准备：数据分析人员或爬虫开发者可将本项目的链接清单作为种子 URL 集合，用于构建垂直领域的内容采集任务。由于链接保持了原始域名和路径结构，便于编写基于域名过滤或路径匹配的采集规则。

开源文档站点的引用索引：开源项目的文档站点往往需要引用大量外部资料。ResourceLink Hub 可作为文档站点的“参考链接”附录，按批次更新并对外公开，既遵守了外部资源的引用规范，又为读者提供了可验证的出处。

## 快速开始

以下步骤适用于 macOS / Linux / Windows WSL 环境，确保系统已安装 Python 3.8 及以上版本和 Git。

```bash
# 克隆仓库到本地
git clone https://github.com/your-org/resourcelink-hub.git
cd resourcelink-hub

# 安装所需依赖（使用 pip 安装到当前用户环境）
pip install --user -r requirements.txt

# 运行本地预览服务（默认监听 8000 端口）
python serve.py --port 8000
```

执行上述命令后，在浏览器中访问 `http://127.0.0.1:8000` 即可查看资源列表首页。如需生成静态站点文件，可执行 `python build.py --output ./dist`，生成的 HTML 文件将输出到 dist 目录，可直接部署到任意 Web 服务器。

## 贡献指南

欢迎外部贡献者提交新的资源链接或改进现有分类。请遵循以下步骤：

1. 复刻本仓库到个人账户，并在本地克隆复刻后的副本。创建新的功能分支，分支名称格式为 `batch-add/描述性前缀`，例如 `batch-add/network-protocols`。

2. 在 `data/batches/` 目录下找到对应批次文件（如需新增批次，请参考 `data-format.md` 中的命名规范）。在批次文件的 `links` 列表末尾追加新链接条目，每个条目需包含 `url` 和 `title` 字段，并确保 `url` 保留原始格式（不添加追踪参数、不转换协议）。

3. 运行 `python scripts/validate_links.py --batch ./data/batches/batch_004.yaml` 对修改后的文件进行格式校验和链接可达性检查。校验通过后，提交变更并推送到复刻仓库。

4. 在本仓库中发起 Pull Request，PR 标题须包含新增链接数量与所属分类。PR 描述中请简要说明每个链接的内容摘要或推荐理由。

5. 项目维护者将在 48 小时内进行审核。审核通过后，PR 将被合并，新增链接随下一轮站点构建自动发布。

## 常见问题

问：部分链接访问时返回 404 或超时，如何处理？

答：本站仅对链接进行索引，不托管源站内容。链接的有效性取决于原始服务器状态。用户可自行重试或通过源站搜索页面查找对应文章。维护者会定期运行 `scripts/validate_links.py` 检测失效链接，并在检测到大量失效时标记该批次为“需审核”状态。如用户发现失效链接，可在 Issue 中附带批次编号和链接地址，维护者将核实后移除或替换。

问：不同批次之间的链接存在重复，如何避免重复收藏？

答：构建流程中的 `scripts/deduplicate.py` 会在每次站点生成时对所有批次进行全局去重扫描。重复链接仅保留首次出现的条目，并在后续批次的页面中显示“此链接已在第 X 批次收录”的提示。贡献者在新增链接时，建议先使用 `scripts/deduplicate.py --check-url <链接>` 查询当前是否已存在相同 URL。

问：为什么每个链接都保留了原始格式，不做统一规范化？

答：本项目坚持“最小干预”原则。原始 URL 中的协议（http 或 https）、主机名、路径大小写以及文件扩展名都可能是源站路由或内容协商的依据。任何修改都可能导致访问失败或返回错误内容。因此，所有链接必须一字不差地原样输出，这是本项目数据准确性的基本承诺。

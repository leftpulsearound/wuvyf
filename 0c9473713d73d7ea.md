# LinkVault Core

LinkVault Core 是一个面向技术研究者、文档维护者和内容聚合者的结构化外链资源索引系统。本项目并非传统的网页爬虫或链接检测工具，而是一个以人工筛选与批量导入相结合的方式，对分散于互联网各处的技术百科、知识库文章进行统一编号、分类存储与快速检索的元数据管理平台。其核心目标在于解决技术资料碎片化、链接失效难追溯以及跨域引用关系混乱等问题，尤其适用于需要长期维护大量外部参考链接的开源文档项目、企业内部知识库或学术文献辅助系统。

本项目当前批次为第 5/56 批，共计收录 180 个经过初步校验的资源链接。系统提供了基于标识符的快速定位、批次状态追踪以及原始链接的只读导出功能，确保用户在不依赖复杂数据库的前提下，仍能对海量外链进行高效的本地化管理。

## 功能概览

**批次化导入引擎** 支持按批次（每批最多 200 条）导入原始 URL 列表，自动解析文件名中的数字标识符，生成内部唯一索引，并记录导入时间戳与批次号。

**原始链接只读存储** 所有用户提供的 URL 均以纯文本形式原样存储于系统内部，禁止任何自动化的协议补全、大小写转换或结尾斜杠添加，保证链接的原始性与可追溯性。

**标识符快速检索** 用户可通过文章详情页中的数字 ID（如 0808、33939）在本地缓存中快速定位对应链接，无需记忆完整 URL 路径。

**批次状态看板** 提供当前批次（5/56）的导入进度、链接总数、重复项检测结果以及与前一批次的增量对比报告。

**裸域名与协议保留策略** 系统严格区分带协议头（http://）与不带协议头的链接，在导出和展示时完全保留用户输入的原始格式，适用于对协议敏感的内网或混合环境。

**Markdown 清单导出** 支持将当前批次的所有链接一键生成为符合项目 README 规范的 Markdown 列表，便于直接粘贴至文档或 wiki 中。

## 应用场景

**技术文档附录管理** 当编写大型开源项目的用户手册或 API 参考文档时，维护者需要引用大量外部百科词条作为背景阅读材料。LinkVault Core 允许将这批 URL 按批次导入，并在文档末尾自动生成格式统一的链接附录，同时保留原始的大小写和协议头，避免因链接格式变更导致的引用失效。

**企业内部知识库外链治理** 企业内部的 Confluence 或 Notion 知识库中往往散落着数百个指向外部供应商、技术论坛或标准规范页面的链接。通过本项目的批次管理功能，知识库管理员可以定期将新增的外链整理为独立批次，并与知识库版本号关联，实现外链的版本化追溯。

**学术论文参考文献辅助** 研究人员在撰写论文或综述时，需要记录大量网络资源的访问路径。LinkVault Core 的原始链接保留策略（不补全 http:// 或 https://）使得参考文献中的 URL 能够严格遵循期刊投稿规范，避免因自动格式化导致的格式错误。

**开源项目 README 资源清单生成** 开源项目的维护者需要定期更新 README 中的相关资源列表。本项目提供了从原始 URL 列表到结构化 Markdown 表格和列表的转换流水线，显著减少手工排版的工作量。

## 快速开始

以下操作将在本地环境中完成 LinkVault Core 的克隆、依赖安装与初始化运行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/linkvault/core.git linkvault-core

# 进入项目根目录
cd linkvault-core

# 安装 Python 依赖（要求 Python 3.8 及以上）
pip install -r requirements.txt

# 执行当前批次（第 5/56 批）的链接导入与索引构建
python cli.py import --batch 5 --source ./data/raw/batch_05.txt

# 启动本地 Web 预览服务（默认端口 8080）
python cli.py serve --port 8080
```

## 项目结构

```
linkvault-core/
├── cli.py                      # 命令行入口，处理导入、检索、导出和服务器启动
├── requirements.txt            # Python 依赖声明（Flask, pytest 等）
├── README.md                   # 项目说明文档（即当前文件）
├── .gitignore                  # Git 忽略规则（缓存、日志、临时文件）
│
├── core/                       # 核心逻辑模块
│   ├── __init__.py             # 模块初始化与公开接口声明
│   ├── importer.py             # 批次导入引擎，解析原始文件并生成索引
│   ├── indexer.py              # 内存索引与持久化缓存管理
│   ├── searcher.py             # 基于标识符的快速检索实现
│   └── exporter.py             # Markdown 列表与 JSON 格式导出器
│
├── data/                       # 数据存储目录（用户输入与系统缓存）
│   ├── raw/                    # 原始输入文件存放处
│   │   └── batch_05.txt        # 当前第 5 批次的原始 URL 列表
│   ├── cache/                  # 索引缓存与元数据持久化
│   │   └── index_v5.json       # 第 5 批次的键值对索引（ID -> URL）
│   └── exports/                # 导出的 Markdown 或 JSON 文件
│       └── batch_05_export.md  # 当前批次的 Markdown 清单导出示例
│
```

## 常见问题

**问：导入时提示 "invalid URL format"，但我的链接看起来是完整的。**  
答：系统默认要求每行必须包含一个合法的 URL 协议头（如 http:// 或 https://），且不能包含空格或不可见字符。请检查原始文件中是否混入了制表符、多余的空格或换行符。如果链接本身是裸域名（如 abc.com），请确保在文件中已明确写出协议头，否则系统会拒绝导入以保持数据格式一致性。

**问：我导入了 180 个链接，但检索时只找到 179 个。**  
答：这通常是由于原始文件中存在完全相同的重复链接，系统在导入时会自动去重并记录警告日志。你可以查看 data/cache/index_v5.json 文件中的 `duplicates` 字段获取被剔除的重复项列表。如果确信没有重复，请检查是否有链接因格式问题被标记为 "skipped"。

**问：如何将当前批次的链接迁移到另一台机器上？**  
答：直接复制整个 data/cache/ 目录到目标机器的相同相对路径下即可。索引文件采用纯 JSON 格式存储，与操作系统和 Python 版本无关。如果目标机器需要重建索引，也可以直接使用 data/raw/batch_05.txt 原始文件重新执行导入命令，系统会覆盖生成一致的索引。

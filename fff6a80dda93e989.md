# LinkVault Core

LinkVault Core 是一个面向技术团队与个人开发者的外链资源归集与导航工具。项目定位于将分散在各类百科、文档、技术博客中的高质量外部链接进行结构化整理，通过统一的索引机制与分类体系，降低技术调研、文档查阅与知识管理中的信息检索成本。本项目适用于需要频繁查阅外部技术资料、维护项目文档链接、或构建团队知识库中资源导航层的场景。

本项目本身不存储任何第三方内容，仅提供链接的元数据描述与分类索引。通过本仓库的目录结构与配置规则，用户可以快速定位到特定主题下的外部资源，同时保持链接的可维护性与可审计性。项目内置了批量链接导入、去重校验、分类标签生成与链接可用性检查等辅助工具，适用于从个人书签管理到企业级文档门户链接治理的多种规模需求。

## 功能概览

**批量链接导入解析**：支持从 CSV、JSON 及 Markdown 列表中批量提取 URL，自动解析域名与路径特征，生成初始分类建议。

**分类标签自动生成**：基于 URL 路径关键词与域名特征，自动为每条链接分配技术领域、内容类型与优先级标签，减少人工分类成本。

**链接可用性巡检**：内置 HTTP 状态检查器，可定期扫描已收录链接的有效性，标记失效链接并生成报告，保障资源列表的可用性。

**多级目录结构映射**：项目目录树与链接分类一一对应，用户可通过文件系统直接浏览分类资源，无需打开数据库或 Web 界面。

**Markdown 原生呈现**：所有资源列表以标准 Markdown 格式维护，与 Git 原生兼容，支持差异对比、历史回溯与协作审阅。

**自定义元数据扩展**：每条链接可附加维护人、添加日期、关联标签与备注说明，元数据存储于独立配置文件中，不污染链接列表本身。

**快速检索与过滤**：提供命令行工具与 VS Code 插件配置，支持按标签、域名或关键词对链接列表进行实时过滤与查找。

**导出与集成支持**：可将链接列表导出为 JSON、HTML 或 OPML 格式，便于导入到浏览器书签、RSS 阅读器或团队文档平台。

## 应用场景

技术团队文档库的外部参考管理。当团队维护内部技术文档或设计文档时，往往需要引用大量外部规范、教程或工具官网。LinkVault Core 可作为文档库的子模块，集中管理这些外部引用，避免链接散落在文档正文中导致维护困难。

个人开发者的技术书签归档。开发者可将日常阅读的技术文章、API 参考、开源项目首页等链接按主题导入项目，利用分类标签与目录结构建立个人知识索引，替代浏览器书签的扁平化管理方式。

开源项目 README 的资源附录生成。开源项目维护者可使用本项目整理与项目相关的社区教程、插件列表、生态工具等外部链接，生成规范的资源附录章节，提升项目的可发现性与生态完整性。

技术调研阶段的链接收集与评估。在进行技术选型或竞品分析时，调研人员可快速将收集到的链接导入系统，统一添加评估备注、优先级标签与分类，形成结构化的调研素材库。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```bash
# 克隆仓库到本地
git clone https://github.com/example/linkvault-core.git
cd linkvault-core

# 安装依赖（项目基于 Python 3.9+ 与 pip）
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行导入示例数据并生成索引
python linkvault.py --import samples/links.csv --output index.md
python linkvault.py --check --timeout 5
```

执行完毕后，`index.md` 文件将包含分类后的链接列表，`reports/` 目录下会生成可用性检查报告。

## 常见问题

**链接导入时出现编码错误如何解决？**

导入 CSV 或 JSON 文件时，请确保文件编码为 UTF-8 且不含 BOM。若使用 Excel 编辑 CSV，建议另存为 UTF-8 编码（无签名）格式。在命令行中可通过 `--encoding utf-8-sig` 参数自动处理 BOM 头。如果仍然遇到解码错误，可使用 `iconv` 或 `recode` 工具转换文件编码后再导入。

**如何自定义链接分类规则？**

分类规则定义于 config.yaml 文件的 `classification.rules` 节点下。每条规则包含 `pattern`（正则表达式，匹配 URL 路径或域名）、`category`（目标分类名称）与 `priority`（优先级，数值越大优先级越高）。修改后无需重启服务，下次运行分类器时自动生效。建议在修改后使用 `--dry-run` 模式预览分类结果，确认无误再正式运行。

**链接可用性检查总是超时或误报怎么办？**

可调整 config.yaml 中的 `checker.timeout`（默认 5 秒）与 `checker.retries`（默认 2 次）参数。对于已知响应较慢的站点，可单独在 `checker.whitelist` 中为其设置更长的超时时间。若站点使用 CloudFlare 或类似的 DDoS 防护机制，建议启用 `checker.follow_redirects` 并设置 `checker.verify_ssl` 为 false（仅在信任内网环境时使用）。检查日志位于 reports/unavailable_links.log，可据此判断是网络问题还是目标站点故障。

# LinkHarbor 技术资源导航站

LinkHarbor 是一个面向开发者、技术研究人员与内容创作者的轻量级技术资源外链汇总平台。项目定位于对互联网公开技术文档、百科条目、开发教程与工程实践文章进行结构化索引与分类导航，帮助用户快速定位到高质量的技术参考资料，减少信息检索过程中的时间损耗与噪声干扰。

本项目不生产内容，而是通过人工筛选与社区贡献的方式，将散落在网络各处的优质技术文章、百科词条、规范文档与工程案例按照主题领域进行归集，形成可长期维护的静态资源导航体系。LinkHarbor 适用于个人知识管理、团队技术文档引用、开源项目参考附录以及技术博客的外部链接补充等场景。

## 功能概览

**按领域分类索引**：所有外链资源按照编程语言、框架生态、基础设施、算法理论、工程实践等一级分类进行组织，每个分类下支持二级标签过滤。

**URL 直链存储**：每条资源记录保存原始 URL、标题、摘要描述与收录时间，不进行二次跳转或短链包装，保证链接可追溯性与原始访问体验。

**资源状态检测**：内置定期链接可达性检查机制，对返回 4xx 或 5xx 状态码的失效链接进行标记并移入待审核队列，维护整体资源库的健康度。

**全文检索与过滤**：基于标题与摘要字段提供关键词快速检索，支持按分类、标签、收录批次等多维度组合过滤，适配不同精度的查找需求。

**批量导入与导出**：支持通过 CSV 或结构化 Markdown 列表批量导入新资源，同时支持将当前索引库导出为 JSON 或 Markdown 格式，便于迁移或备份。

**社区贡献工作流**：提供基于 Pull Request 的资源新增、更新与删除流程，每位贡献者的提交记录均留痕，保证资源库演进的透明性与可审计性。

## 应用场景

技术团队内部知识库建设：开发团队可将 LinkHarbor 部署为内部文档站点的外链附录，集中管理项目依赖的技术规范、第三方库文档、运维手册与故障排查案例，减少成员在多个平台间反复切换查阅的碎片化体验。

开源项目 README 与文档引用：开源维护者可以在项目的 README 中引用 LinkHarbor 中归集的参考链接，替代直接在文档中罗列大量裸 URL 的做法，提升文档的可读性与维护性，同时便于统一更新失效引用。

技术博客与教程的外部链接附录：技术博主或在线课程作者可将 LinkHarbor 作为文章或课程章节的扩展阅读索引，为读者提供经过筛选的深度学习材料，避免因个人博客链接变动导致引用断裂。

个人技术阅读清单管理：开发者可将 LinkHarbor 作为个人技术阅读的收藏夹替代方案，通过分类和标签体系管理自己关注的技术领域资源，定期回顾与清理，形成可持续迭代的知识体系。

## 快速开始

以下命令演示了如何从 GitHub 克隆项目、安装依赖并启动本地开发服务器。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkharbor.git

# 进入项目目录
cd linkharbor

# 安装依赖（使用 npm）
npm install

# 启动本地开发服务，默认监听端口 3000
npm run dev
```

启动成功后，访问控制台输出的本地地址即可进入资源管理面板。首次启动会自动创建 SQLite 数据库文件并初始化分类与标签表结构。

## 安装要求

| 依赖 | 必需 | 说明 |
|---|---|---|
| Node.js 18.x 或更高版本 | 是 | 运行时环境，用于执行服务端代码与构建脚本 |
| npm 9.x 或 yarn 1.22+ | 是 | 包管理器，用于安装项目依赖 |
| SQLite 3.39+ | 是 | 嵌入式关系型数据库，用于存储资源索引与元数据 |
| Git 2.30+ | 是 | 版本控制系统，用于克隆仓库与管理贡献变更 |
| PM2 5.x（生产环境推荐） | 否 | 生产环境进程守护工具，增强服务稳定性 |
| Nginx 或 Caddy（生产环境推荐） | 否 | 反向代理服务器，用于处理静态资源缓存与 HTTPS 终结 |

## 贡献指南

我们欢迎并鼓励社区贡献者参与 LinkHarbor 资源库的建设与维护。所有贡献需遵循以下工作流程。

第一，在 GitHub 上 Fork 本项目到个人账户，然后克隆到本地开发环境。请确保本地 Node.js 与 SQLite 版本满足安装要求，并成功执行快速开始步骤中的三条命令完成环境搭建。

第二，新增资源链接时，在项目根目录的 /data/pending/ 下创建以批次号命名的 JSON 或 Markdown 文件，按照模板填写 URL、标题、分类标签与简短摘要。提交前请运行 npm run validate 对新增链接进行格式与可达性预检，确保未包含重复或失效地址。

第三，更新或删除已有资源时，直接在 /data/index/ 对应的分类文件中修改记录，并更新 updated_at 时间戳。对于失效链接，建议先标记为 status: "unreachable" 而非直接删除，以便后续维护者复核。

第四，所有变更通过 Pull Request 提交，PR 标题需包含变更类型前缀（[add]、[update]、[remove]、[fix]），并在描述中说明变更原因与测试结果。PR 至少需要一位项目维护者审核通过后方可合并入主分支。

第五，合并后，维护者会执行 npm run rebuild 重新构建完整索引，并触发一次全量链接健康检查。贡献者信息将自动记录在 CONTRIBUTORS.md 文件中，按 PR 合并时间排序。

## 常见问题

问：我发现某个收录链接已经失效，应该如何处理？

答：您可以直接在项目的 Issues 中提交失效链接报告，标题格式为 [Broken Link] + 原始 URL，并在描述中附带访问时返回的 HTTP 状态码。维护者会在收到报告后的五个工作日内进行复核，确认失效后将链接移入待审核区。您也可以按照贡献指南中的流程提交 PR 直接标记该链接为失效状态。

问：如何申请新增一个分类或标签？

答：分类与标签的增删属于结构性变更，涉及数据库 schema 与前端过滤逻辑。建议先在 Discussions 中发起提案，说明新增分类的名称、覆盖范围与预期收录的资源数量，获得至少两位社区成员赞同后，由维护者执行变更。不鼓励贡献者在 PR 中直接新增分类，以免造成分类体系的碎片化。

问：本项目是否可以部署到内网环境，不访问公网？

答：可以。LinkHarbor 的核心功能不依赖任何外部 API 或 CDN 资源，所有依赖项均在 npm 安装时完整下载。您只需将本仓库完整克隆到内网 Git 服务器，然后在内网执行 npm install 与 npm run build 即可完成离线部署。链接健康检查功能需要访问公网以验证外链状态，若在内网使用，可关闭该功能或配置代理。

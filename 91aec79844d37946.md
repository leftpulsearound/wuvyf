# WapWiki Link Aggregator

WapWiki Link Aggregator 是一个面向技术研究者和信息分析人员的结构化外链汇总与百科条目索引系统。该项目将分散在移动端百科站点中的高质量技术文章、概念解析和案例文档进行集中式收录与分类管理，解决信息碎片化导致的检索效率低下问题。通过提供统一的条目导航和本地化镜像访问方案，该项目适用于需要快速查阅特定技术术语、历史版本记录或跨领域知识关联的场景。

目标用户包括开源社区维护者、技术文档撰写人、软件开发工程师以及计算机科学领域的研究生。本项目不对原始内容进行修改或翻译，仅提供索引与结构化呈现层，确保信息来源的可追溯性与完整性。当前批次为第 35/56 批，累计处理资源链接 180 条。

## 功能概览

批量条目导入 支持基于文章ID的自动抓取与元数据提取，可识别标题、分类、发布时间及正文摘要。

分类标签系统 为每个收录条目自动生成层级标签，支持按技术领域、难度等级、内容类型进行筛选。

全文检索接口 提供基于关键词的站内搜索能力，支持布尔运算符与模糊匹配，便于在大量条目中定位特定信息。

本地缓存镜像 对原始链接内容进行静态化备份，降低源站访问波动对阅读体验的影响，同时保留原始 URL 作为永久引用地址。

变更追踪日志 记录每次条目更新或新增的时间戳与操作类型，支持导出为结构化日志文件用于审计。

导出与分享 支持将选中的条目列表导出为 Markdown、JSON 或 CSV 格式，便于集成到其他知识管理工具中。

访问统计看板 展示条目被查阅的频率排行、分类热度分布以及来源域名请求占比，帮助识别高价值内容区域。

## 应用场景

技术团队内部知识库建设 团队可以将本项目的条目索引作为基础层，结合内部文档形成闭环的知识体系。例如，在排查特定框架版本兼容性问题时，可通过本索引快速跳转至相关的百科解释原文。

学术研究文献溯源 研究人员在撰写综述或技术报告时，需要引用权威的在线定义。本项目的固定条目结构可作为稳定的引用锚点，避免因原站内容调整导致参考文献失效。

离线环境下的备查工具 在无法直接访问外网的生产环境中，运维人员可利用本系统的本地缓存功能预先同步关键条目，构建内部可用的技术词典镜像。

个人学习路径规划 开发者可按分类标签筛选出一系列相关概念条目，形成从基础到进阶的阅读清单，用于系统化补充特定方向的理论知识。

## 快速开始

以下步骤适用于 Linux/macOS 环境，Windows 用户可使用 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库
git clone https://github.com/example/wapwiki-link-aggregator.git
cd wapwiki-link-aggregator

# 安装依赖（使用 Python 3.9+）
pip install -r requirements.txt

# 运行本地索引服务（默认监听 127.0.0.1:8000）
python serve.py --port 8000
```

启动后，访问 http://127.0.0.1:8000 即可查看当前批次已导入的条目列表。首次运行时会自动执行数据库迁移并生成示例分类标签。如需同步最新条目，可使用 `python sync.py --batch 35` 命令从指定数据源拉取更新。

## 项目结构

```
wapwiki-link-aggregator/
├── serve.py                       # 主服务入口，启动 HTTP 服务与路由注册
├── sync.py                        # 同步脚本，执行批次拉取与增量更新逻辑
├── requirements.txt               # Python 依赖清单，固定版本号
├── config/
│   ├── settings.yaml              # 全局配置，包含端口、缓存路径、超时参数
│   └── sources.json               # 数据源定义，记录基础域名与URL模板
├── core/
│   ├── fetcher.py                 # 内容抓取模块，封装 requests 与重试策略
│   ├── parser.py                  # HTML 解析器，基于 BeautifulSoup 提取字段
│   ├── indexer.py                 # 索引管理，负责 SQLite 的读写操作
│   └── validator.py               # 链接可用性校验，检测 HTTP 状态码
├── storage/
│   ├── database/                  # SQLite 数据库文件目录
│   │   └── entries.db             # 主数据库，包含 articles 与 tags 表
│   └── cache/                     # 静态缓存目录，按文章 ID 存储 .html 副本
├── tests/
│   ├── test_fetcher.py            # 抓取模块的单元测试
│   ├── test_parser.py             # 解析模块的测试用例
│   └── test_integration.py        # 端到端集成测试
├── docs/                          # 文档目录，包含用户手册与开发指南
│   ├── usage.md
│   ├── admin.md
│   ├── development.md
│   └── contributing.md
└── scripts/
    ├── export_json.py             # 导出为 JSON 格式的辅助脚本
    └── clean_cache.sh             # 清理过期缓存的 Shell 工具
```

## 贡献指南

我们欢迎社区开发者提交改进建议与代码补丁。请遵循以下流程以确保贡献被顺利接收。

1. 查阅问题追踪列表 在提交新功能或修复之前，请先浏览 GitHub Issues 中已有的讨论，避免重复劳动。如果发现尚未记录的问题，请创建一个新的 Issue 并附上复现步骤。

2. 创建功能分支 基于 main 分支创建以你的功能或修复命名的分支，例如 feat/add-filter-by-category 或 fix/parser-encoding-issue。分支名称应简洁描述变更内容。

3. 编写或更新测试 所有新功能必须包含对应的单元测试，测试覆盖率不得低于 80%。修复缺陷时，请添加一个能够暴露该缺陷的测试用例，确保修复后测试通过。

4. 提交拉取请求 在拉取请求描述中详细说明变更动机、实现方法以及可能的副作用。如果涉及文档更新，请同步修改 docs/ 目录下的对应文件。

5. 代码审查与合并 至少一位项目维护者将审查你的提交。审查通过后，由维护者执行 squash 合并，并更新 CHANGELOG.md 中的版本记录。

## 常见问题

如何添加自定义数据源或修改解析规则？

本项目设计为可扩展架构。你可以在 config/sources.json 中注册新的域名模板，并在 core/parser.py 的 ParserRegistry 类中注册对应的解析函数。解析函数需要接收原始 HTML 字符串，返回包含 title、content、tags 的标准字典结构。详细说明请参考 docs/development.md 中的适配器开发章节。

为什么某些链接在本地缓存中显示为不可访问？

本项目仅对原始链接进行被动缓存，不保证源站始终可用。如果某个条目返回 HTTP 4xx 或 5xx 状态码，系统会在日志中记录错误并跳过该次缓存更新。此时条目索引仍然保留，但预览功能可能不可用。你可以通过 `sync.py --retry-failed` 命令重新尝试获取失效链接的内容。

如何将本系统部署到生产环境（如 Docker）？

项目根目录提供了 Dockerfile 和 docker-compose.yaml 示例文件。你只需要执行 `docker-compose up -d` 即可启动包含 Nginx 反向代理和 Gunicorn 应用服务器的容器集群。生产环境建议将 SQLite 数据库替换为 PostgreSQL，并配置环境变量 `DATABASE_URL` 来切换后端存储。更详细的部署方案请查阅 docs/admin.md 中的生产部署章节。

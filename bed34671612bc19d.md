# ResourceHub

ResourceHub 是一个面向开发者和技术研究人员的结构化技术资源导航与百科外链聚合平台。本项目不生产原始内容，而是通过人工筛选与自动化校验，将分散于互联网各处的优质技术百科条目、开发文档、问题排查记录与最佳实践文章按主题域进行归集，并提供统一的访问入口与元数据标注。

ResourceHub 定位于技术团队的日常查阅工具与知识中台辅助组件，解决开发者在面对海量技术信息时检索成本高、来源质量参差不齐、缺乏上下文关联等核心痛点。通过本项目提供的资源索引体系，用户可快速定位到特定技术栈的权威解释、配置范例或故障处理参考，显著提升问题解决效率。

## 功能概览

**多维度资源索引**：按技术栈、应用场景、文档类型与时效性对收录的 URL 进行标签化分类，支持快速筛选。

**百科条目聚合**：集中展示来自同一信源的不同条目，便于对比阅读与交叉验证。

**结构化元数据提取**：对每条资源自动标注标题推测、内容摘要与关键词，辅助用户在不打开页面前预判内容价值。

**外链健康度监控**：定期对收录 URL 进行可达性检查，标记失效链接并生成报告。

**自定义收藏与笔记**：允许用户对特定条目添加个人标签与备注，构建个性化知识库。

**全文检索支持**：基于标题与元数据字段提供轻量级搜索接口，支持模糊匹配与布尔组合查询。

## 应用场景

技术团队在日常开发中遭遇第三方库兼容性问题或运行时异常时，可通过 ResourceHub 检索相关错误码或组件名称，快速获取经过初步筛选的百科条目与社区讨论链接，缩短排障时间。

技术培训负责人为新员工制定入门学习路径时，可依据 ResourceHub 的分类导航，按语言、框架、数据库等维度组织阅读清单，确保学习资料的系统性与覆盖面。

个人开发者在对某项新兴技术进行预研评估时，可利用 ResourceHub 的聚合视图，快速浏览该领域的核心概念解释、典型配置案例与已知限制说明，辅助技术选型决策。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash。

```bash
# 克隆仓库
git clone https://github.com/your-org/resourcehub.git
cd resourcehub

# 安装依赖（Python 3.9+ 环境）
pip install -r requirements.txt

# 初始化本地索引并启动开发服务器
python manage.py migrate
python manage.py runserver
```

生产环境部署请参考 `docs/deployment.md`，推荐使用 Gunicorn + Nginx 组合。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Python | 3.9 - 3.11 | 核心运行时，用于后端服务与索引脚本 |
| Django | 4.2.x | Web 框架，提供管理后台与 API 端点 |
| SQLite / PostgreSQL | SQLite 3.31+ / PostgreSQL 13+ | 元数据存储，开发环境默认使用 SQLite |
| redis | 6.0+ | 缓存与任务队列后端，非必需但推荐 |
| nodejs | 16.x+ | 仅用于前端资源构建（可选） |
| curl / wget | 最新稳定版 | 用于健康度检查脚本的外部调用 |

## 项目结构

```
├── manage.py                       # Django 项目管理入口
├── requirements.txt                # Python 依赖清单
├── config/                         # 项目全局配置
│   ├── settings.py                 # 基础配置（含数据库、缓存、中间件）
│   ├── settings_dev.py             # 开发环境覆盖配置

├── apps/                           # 所有应用模块
│   ├── core/                       # 核心数据模型与抽象基类
│   │   ├── models.py               # Resource, Category, Tag, HealthCheck 模型
│   │   └── managers.py             # 自定义查询管理器
│   ├── crawler/                    # 资源抓取与元数据提取
│   │   ├── fetcher.py              # 异步 HTTP 请求与重试逻辑
│   │   ├── parser.py               # 基于 BeautifulSoup 的标题与摘要提取
│   │   └── tasks.py                # Celery 周期性任务（健康度检查）
│   ├── api/                        # RESTful API 接口
│   │   ├── views.py                # 资源列表、详情、搜索端点
│   │   ├── serializers.py          # 数据序列化与校验
│   │   └── filters.py              # 自定义过滤与排序
│   ├── users/                      # 用户认证与个人收藏
│   │   ├── auth.py                 # JWT 认证后端
│   │   └── profiles.py             # 用户笔记与标签扩展
│   └── dashboard/                  # 管理后台（Django Admin 扩展）
│       ├── admin.py                # 模型管理注册与列表显示定制
│       └── widgets.py              # 自定义表单控件
├── static/                         # 静态文件（CSS / JS / 图标）
│   ├── css/
│   └── js/
├── templates/                      # 服务端渲染模板（仅用于错误页与管理界面）
│   ├── 404.html
│   └── 500.html
├── docs/                           # 文档（见上方文档导航）
│   ├── user-guide/
│   ├── admin-guide/
│   ├── dev-guide/
│   └── ops-guide/
├── scripts/                        # 运维与辅助脚本
│   ├── health_check.py             # 独立健康度检测脚本
│   ├── import_links.py             # 批量导入 URL 清单
│   └── export_index.py             # 导出索引为 JSON / CSV
├── tests/                          # 单元测试与集成测试
│   ├── test_models.py
│   ├── test_api.py
│   └── test_parser.py
└── .env.example                     # 环境变量模板（数据库连接、密钥、缓存地址）
```

## 贡献指南

我们欢迎各类贡献，包括但不限于新增资源条目、改进元数据解析逻辑、完善文档以及提交缺陷修复。请遵循以下步骤：

1. 查阅 `docs/dev-guide/` 下的开发环境搭建指南，确保本地开发环境满足所有依赖要求，并运行测试套件确认基线通过。

2. 在 `issues` 页面搜索是否存在相关议题，若无则新建议题简要描述您拟解决的问题或拟添加的资源范围，等待维护者标注 `accepted` 标签后再开始编码。

3. 从 `main` 分支创建新的特性分支，命名格式为 `feature/xxx` 或 `fix/xxx`，遵循 PEP 8 编码规范并为新增代码补充单元测试，确保测试覆盖率达到 80% 以上。

4. 提交代码时使用清晰的中文或英文 commit 信息，格式为 `<类型>: <简短描述>`，例如 `feat: 增加按语言过滤资源的 API 参数`。

5. 发起 Pull Request 至 `main` 分支，在描述中关联对应的议题编号，并简要说明测试结果与变更影响范围。等待至少一名维护者进行 Code Review，并根据反馈完成修订后合入。

## 常见问题

**Q: 收录的资源链接出现无法访问时，平台如何处理？**

A: 健康度监控任务每 24 小时执行一次，对每条资源发起 HEAD 请求。连续三次检测不可达的链接将被标记为 `unreachable`，并在管理后台高亮显示。已标记的链接不会从索引中删除，但会在前端检索结果中降权排序，并附加警告标识。用户可手动提交复检请求。

**Q: 如何请求添加新的资源来源或批量提交链接？**

A: 个人用户可通过页面底部的「提交建议」表单单条提交。团队或批量提交需求，请参考 `scripts/import_links.py` 脚本的说明，按指定 JSON 格式组织链接列表，并在 GitHub 议题中上传样例文件，维护者会在评估后导入。我们不接受包含自动跳转或强制下载的链接。

**Q: 本项目与通用搜索引擎的区别是什么？**

A: 通用搜索引擎返回的结果包含大量广告、低质聚合页与时效性不明的内容。ResourceHub 仅收录来自同一稳定信源的结构化百科条目，并通过人工规则过滤了非技术类页面。每个链接都带有主题预测标签与初步摘要，更适用于快速查阅概念定义、参数含义和典型配置，而非泛化搜索。

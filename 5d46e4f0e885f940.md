# ResourceBridge

ResourceBridge 是一个面向技术团队与个人开发者的外链资源归集与导航系统。该项目并非传统的内容聚合平台，而是一个结构化的外部知识索引库，专注于对分散在互联网各处的技术文档、百科条目、工具站点及社区讨论进行系统性整理与分类。通过标准化的元数据提取与分类模型，ResourceBridge 帮助用户从庞杂的信息噪音中快速定位高价值的技术参考链接，降低跨站点检索的时间成本。

项目定位为“技术决策的知识缓存层”，适用于需要频繁查阅外部技术资料的架构师、运维工程师与研发管理者。ResourceBridge 不存储文章正文，仅保留链接地址、摘要标签与关联关系，所有原始内容仍由源站提供，项目本身只承担索引与导航职能。

## 功能概览

- **多源链接归集**：支持批量导入外部 URL，自动解析域名与路径结构，按来源站点与内容类型生成初步分类标签。

- **自定义标签体系**：用户可为每条链接添加多级标签（如“运维/监控/日志”），支持标签组合检索与快速筛选。

- **全文元数据检索**：基于链接标题、摘要、标签及来源域名的全文搜索，支持模糊匹配与精确过滤。

- **批次管理**：内置批次导入机制，每一批新增链接均可独立标记与追溯，便于团队协作审阅与增量更新。

- **访问状态监测**：定期对已收录链接进行 HTTP 状态检查，标记失效或重定向链接，辅助维护索引健康度。

- **外链导出**：支持将筛选后的链接列表导出为 Markdown、JSON 或 CSV 格式，用于集成到其他文档或监控系统。

- **阅读标记**：用户可对已查阅的链接添加“已读”“待整理”“高价值”等状态标记，支持个人进度跟踪。

- **暗色主题与移动端适配**：前端界面提供明暗两种主题，并对手机浏览器进行布局优化，满足移动端查阅需求。

## 应用场景

- **技术选型调研**：团队在引入新中间件或框架时，可将候选产品的官方文档、性能评测帖、社区问答等链接统一录入 ResourceBridge，按“选型/对比/案例”等标签归类，便于横向对比与会议评审。

- **运维故障知识库**：运维人员可将生产环境中遇到的异常报错对应的解决方案链接、官方 issue 讨论、临时修复脚本地址等集中收录，形成内部可共享的故障排查索引，加速后续同类问题的定位。

- **新人入职学习路径**：为新入职的研发人员定制学习路线图，将内部培训文档、外部最佳实践指南、API 参考手册等链接按阶段分类，通过 ResourceBridge 生成有序的学习清单，减少新人摸索时间。

- **开源项目依赖跟踪**：开源项目维护者可将依赖库的发布公告、CVE 漏洞报告、版本迁移指南等链接统一管理，便于版本升级前进行影响面评估。

## 快速开始

以下步骤适用于 Linux / macOS 系统，需提前安装 Git 与 Node.js（v18 及以上）。

```bash
# 克隆仓库
git clone https://github.com/resourcebridge/resourcebridge.git

# 进入项目目录
cd resourcebridge

# 安装依赖（使用 npm）
npm install

# 以开发模式启动（默认监听端口 3000）
npm run dev
```

启动后，在浏览器中访问 http://localhost:3000 即可进入 ResourceBridge 的仪表盘界面。首次启动将自动创建 SQLite 数据库文件 `data/resourcebridge.db`，无需额外配置数据库服务。

## 安装要求

| 依赖组件 | 必需版本 | 说明 |
|---------|---------|------|
| Node.js | 18.x 或 20.x LTS | 运行时环境，推荐使用 nvm 管理版本 |
| npm | 9.x 或以上 | 包管理器，用于安装项目依赖 |
| SQLite | 3.35 或以上（内置） | 项目内嵌轻量级数据库，无需单独安装 |
| Git | 2.30 或以上 | 用于克隆仓库及版本管理 |
| 可用磁盘空间 | 至少 200 MB | 用于存放数据库索引及临时文件，链接数量增加时建议预留 1 GB |
| 操作系统 | Linux / macOS / Windows WSL2 | 生产部署建议使用 Ubuntu 20.04 或更新版本 |
| 现代浏览器 | Chrome 100+ / Firefox 100+ / Edge 100+ | 前端界面需支持 ES2020 语法 |
| 网络访问 | 外网出站可达（用于状态监测） | 若部署在内网，可关闭状态监测功能 |

## 项目结构

```
resourcebridge/
├── backend/                        # 后端服务目录
│   ├── src/
│   │   ├── controllers/            # 请求控制器，处理链接CRUD与批次管理
│   │   ├── models/                 # 数据模型定义（链接、标签、批次、用户状态）
│   │   ├── services/               # 业务逻辑层（解析器、状态监测、导出引擎）
│   │   ├── middleware/             # 鉴权、日志、错误处理中间件
│   │   ├── routes/                 # RESTful API 路由定义
│   │   ├── workers/                # 后台任务（定时状态检查、批次导入队列）
│   │   └── utils/                  # 工具函数（URL标准化、摘要生成、日期处理）
│   ├── tests/                      # 单元测试与集成测试用例
│   └── package.json                # 后端依赖声明（Express、SQLite3、Axios）
├── frontend/                       # 前端单页应用目录
│   ├── src/
│   │   ├── pages/                  # 页面级组件（仪表盘、链接列表、批次详情、设置）
│   │   ├── components/             # 可复用UI组件（标签选择器、链接卡片、状态徽章）
│   │   ├── hooks/                  # 自定义React Hooks（数据获取、本地存储）
```

## 常见问题

**Q：收录的链接数量超过一万条时，页面加载变慢甚至超时，应如何优化？**

A：ResourceBridge 默认对链接列表采用分页加载策略，每页默认 50 条。若仍感觉响应缓慢，建议在 `backend/src/services/query.js` 中调整索引字段，为 `url` 和 `tag` 列创建额外的 SQLite 索引。此外，可在环境变量中将 `PAGE_SIZE` 调低至 20 或 30，减少单次查询的数据量。对于超过五万条链接的实例，推荐将 SQLite 数据库迁移至 PostgreSQL 以获得更好的并发查询性能。

**Q：导入 URL 时能否自动提取文章标题和摘要，而非手动填写？**

**A：** 可以。ResourceBridge 内置了基于 `cheerio` 的轻量级 HTML 解析器。在导入时若仅提供 URL 而未填写标题，系统会自动请求该链接并解析 `<title>` 标签内容作为标题，同时提取 `meta[name=description]` 的内容作为摘要。若目标站点禁止爬取或返回非 HTML 内容（如 PDF、图片），则回退为使用 URL 路径末尾的字段作为默认标题。该功能默认启用，可在 `backend/src/services/fetcher.js` 中关闭自动抓取选项。

**Q：项目是否支持多用户协同管理？**

**A：** 当前版本（v1.x）聚焦于单用户或共享只读视图模式，默认使用单一 SQLite 文件存储，不内置用户认证体系。若团队需要多用户编辑与权限控制，建议在前端部署时增加反向代理层（如 OAuth2 Proxy），在 HTTP 头部注入用户标识，ResourceBridge 会读取 `X-Forwarded-User` 头作为用户区分字段，实现基础的隔离记录。完整的 RBAC 功能已在 v2.0 路线图中，预计支持管理员、编辑者、访客三级权限。
